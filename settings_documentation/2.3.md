### Для пункта 2.3

Кратко обоснуй выбор средств разработки для информационной системы электронного расписания. Покажи, почему выбранный стек (PHP, MySQL, Apache, Bootstrap) подходит для образовательной организации и задач централизованного хранения, отображения и актуализации расписания.

### Для пункта 2.3.1

Опиши требования к информационным структурам и методам решения:
- какие сущности должны храниться в базе данных: занятия (schedule_all), настройки расписания (schedule_settings), пользователи (users), статистика просмотров групп (group_activity);
- поля занятия: группа, номер недели, день недели, номер пары, дисциплина, аудитория, преподаватель, время начала и окончания;
- почему для этих данных целесообразна реляционная модель;
- как клиент-серверный подход обеспечивает разделение интерфейса, бизнес-логики (ScheduleManager) и хранения данных;
- как обеспечиваются целостность, масштабируемость, безопасность, производительность и расширяемость.

### Для пункта 2.3.2

Опиши требования к техническим средствам:
- серверная среда для разработки и эксплуатации (Apache, PHP 8.1+, MySQL 8.0+, XAMPP на этапе разработки);
- минимальные и рекомендуемые параметры серверной части;
- требования к клиентской части (современный браузер, подключение к сети, отсутствие необходимости установки специального ПО на ПК);
- требования к мобильным устройствам для адаптивного просмотра расписания;
- объяснение, почему это повышает доступность системы для администрации, преподавателей и обучающихся.

### Для пункта 2.3.3

Опиши требования к исходным кодам и языкам программирования:
- разделение серверной логики, шаблонов интерфейса, стилей и сценариев;
- применение PHP для серверной части и HTML, CSS, JavaScript, Bootstrap для клиентской;
- требования к качеству кода: понятные имена, структурированность, единый стиль;
- обязательные меры: подготовленные запросы, проверка входных данных, экранирование вывода, CSRF-защита, контроль доступа по ролям;
- использование MySQL с индексами по группе, неделе и дню, уникальным ключом на комбинацию группа-неделя-день-пара;
- контроль версий с фиксацией изменений по функциональным блокам.
